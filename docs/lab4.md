# Аналіз комунікації між компонентами (Лабораторна робота №4)

## 1. Що змінилося порівняно з лабораторною 3?

У третій лабораторній роботі наша система працювала **суворо лінійно** і обробляла лише основну бізнес-логіку (CRUD операції) через Command та Query Handlers.

У цій роботі ми впровадили **побічні ефекти**, які виникають як реакція на основні дії. Ми додали новий допоміжний компонент — `NotificationService`, який ізольований у власному модулі та має чіткий контракт для відправки електронних листів.

Для інтеграції цього сервісу ми реалізували два підходи:

- **синхронний** (через прямий виклик);
- **асинхронний** (через імплементацію власного `EventBus`).

## 2. Які побічні операції ви виділили і чому?

Для домену Task Manager було виділено таку побічну операцію: **відправка Email-сповіщень**.

**Чому?**  
Користувачі повинні отримувати вітальний лист при реєстрації та сповіщення при створенні або виконанні задачі. Ця логіка не є частиною основного домену (збереження задачі в БД), тому її винесено в окремий допоміжний компонент `NotificationService`.

## 3. Порівняння синхронного та асинхронного підходів

Ми застосували **синхронний** підхід для операції реєстрації користувача (`RegisterUserHandler`) та **асинхронний** підхід для операцій із задачами (`CreateTaskHandler`, `UpdateTaskHandler`).

### Час відповіді API

- **Синхронний**: Час відповіді значно збільшується (складає понад 200–1500 мс), оскільки HTTP-запит клієнта блокується, поки `NotificationService` не встановить з'єднання з SMTP-сервером і не відправить лист.
- **Асинхронний**: Час відповіді мінімальний (15–50 мс). Handler просто публікує незмінну (`immutable`) подію `TaskCreatedEvent` у шину повідомлень і миттєво повертає клієнту статус `201 Created`. Відправка листа відбувається у фоні.

### Поведінка при збоях

- **Синхронний**: Якщо поштовий сервіс недоступний, виникає ризик відкочування успішної реєстрації. Виклик сервісу було обгорнуто в `try/catch`. Помилка логується та ігнорується, що дозволяє основній операції успішно завершитись.
- **Асинхронний**: Основна операція на 100% захищена від збоїв допоміжного компонента. Якщо `NotificationSubscriber` падає, `EventBus` перехоплює помилку, не впливаючи на відповідь для клієнта.

### Зв'язаність між компонентами

- **Синхронний**: Висока зв'язаність (**Tight Coupling**). `RegisterUserHandler` напряму знає про існування `NotificationService` (хоча й через інтерфейс).
- **Асинхронний**: Слабко зв'язана архітектура (**Decoupling**). `CreateTaskHandler` взагалі не знає про відправку листів. Він лише фіксує факт події, передаючи "багату" подію з достатнім контекстом. `NotificationSubscriber` слухає цю подію і викликає сервіс.

### Складність реалізації та тестування

- **Синхронний**: Реалізується дуже просто (один рядок коду), але вимагає обережності при тестуванні (потрібно імітувати помилки сторонніх сервісів).
- **Асинхронний**: Складніший в імплементації. Вимагає створення інфраструктури (`EventBus`, підписники, класи подій). Тестування також ускладнюється, оскільки виклики відбуваються у фоновому режимі (`setImmediate` / `setTimeout`).

## 4. Який підхід ви б обрали для production-системи і чому?

Для production-системи я б **однозначно обрав асинхронний підхід**.

**Чому?**  
У високонавантажених (high-load) системах ми не можемо змушувати користувача чекати на виконання важких або ненадійних I/O операцій (робота з мережею, SMTP тощо). Асинхронність через зовнішній брокер повідомлень (RabbitMQ, Kafka або Redis) забезпечує масштабованість, відмовостійкість і гарантує, що повідомлення не буде втрачено навіть при перезавантаженні сервера.

## 5. Ідемпотентність: Що станеться, якщо подія буде доставлена двічі?

У поточній реалізації (in-process Event Bus) події не дублюються. Однак, якщо ми перейдемо на зовнішній брокер повідомлень у production, існує ймовірність дублювання (принцип **"at-least-once delivery"**).

- **Наслідок зараз**: Якщо подія `TaskCreatedEvent` прийде двічі, користувач отримає два однакових електронних листи. Наші обробники наразі не є ідемпотентними.
- **Як виправити в production**: Необхідно додати таблицю `ProcessedEvents` у базу даних. При отриманні події підписник спочатку перевіряє `eventId`. Якщо подія з таким ID вже була успішно оброблена — вона ігнорується. Це зробить обробник ідемпотентним і захистить систему від дублювання побічних ефектів.
