# Аналіз впровадження шарової архітектури

## 1. Що змінилося в структурі проєкту порівняно з лабораторною 1?

У першій лабораторній роботі проєкт мав **зв’язану структуру** («все в купі»), де контролери часто напряму викликали ORM (Prisma) або містили бізнес-логіку всередині себе.

Тепер систему чітко розділено на **4 незалежні шари**:

- **Presentation Layer** — відповідає лише за прийом HTTP-запитів та повернення відповідей.
- **Application Layer** — містить Use Cases (реєстрація, створення задачі) та оркеструє виклики до домену і бази даних.
- **Domain Layer** — серце системи. Тут живуть чисті бізнес-моделі (`User`, `Task`), Value Objects та Domain Factory. Цей шар **не має жодних імпортів** фреймворку чи ORM.
- **Infrastructure Layer** — містить реалізацію роботи з базою даних (Prisma), мапери та зовнішні сервіси.

## 2. Які переваги дало розділення на шари?

- **Ізоляція бізнес-логіки**: Усі бізнес-правила (наприклад, перевірка, що дата дедлайну задачі не може бути в минулому) тепер живуть виключно в домені. Вони захищені від змін у базі даних чи HTTP-фреймворку.
- **Інверсія залежностей (DIP)**: Домен диктує правила (інтерфейси репозиторіїв), а інфраструктура під них підлаштовується (адаптери).
- **Справжні Unit-тести**: Тепер можна тестувати поведінку `Task` або створення об’єктів через Domain Factory миттєво, без необхідності піднімати PostgreSQL або мокати Prisma.

## 3. Які недоліки / ускладнення з'явилися?

- **Збільшення об’єму коду (Boilerplate)**: Для передачі даних тепер використовуються DTO.
- **Необхідність мапінгу**: Оскільки доменна модель (`Task`) та модель бази даних (Prisma `Task`) — це різні класи з різними відповідальностями, довелося написати спеціальні мапери (Data Mappers) у шарі інфраструктури.
- **Складніша навігація**: Щоб відслідкувати один HTTP-запит, тепер потрібно пройти через 4 різні папки.

## 4. Наскільки простіше тепер змінити БД або фреймворк?

Змінити технологічний стек стало **максимально просто** завдяки правилу залежностей:

- **Зміна БД**: Якщо ми вирішимо відмовитися від Prisma + PostgreSQL на користь TypeORM + MySQL або навіть MongoDB, достатньо переписати класи лише у шарі **Infrastructure**. Домен і Use Cases навіть не дізнаються про цю зміну, оскільки вони спілкуються з БД через абстрактні інтерфейси (порти).
- **Зміна фреймворку**: Заміна Express на Fastify вплине лише на шар **Presentation**.

## 5. Чому було обрано Rich Domain Model (Багату доменну модель)?

Для системи Task Manager було обрано підхід **Rich Domain Model**.

**Обґрунтування:**  
Наші сутності не є просто «мішками з даними» (Anemic Domain Model). Згідно з юзкейсами, сутність `Task` має захищати свої інваріанти: наприклад, вона не може бути створена з датою дедлайну в минулому, а її статус має змінюватися лише за певними правилами (`PENDING → IN_PROGRESS → COMPLETED`).

**Реалізація:**  
Усі перевірки та методи зміни стану (наприклад, `task.markAsCompleted()`) інкапсульовані всередині самого класу `Task`. Це гарантує, що жоден інший сервіс не зможе перевести задачу в невалідний стан.  
Для створення нових задач зі складнішими валідаціями використовується **Domain Factory**.
