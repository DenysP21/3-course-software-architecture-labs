# Аналіз впровадження CQS (Лабораторна робота №3)

## 1. Що змінилося в структурі проєкту порівняно з лабораторною 2?

Головна зміна полягає у відмові від монолітних сервісів (`UserService`, `TaskService`) на користь спеціалізованих об'єктів.

- Було видалено папку `src/services`.
- З'явилися нові папки `src/commands` (для операцій запису) та `src/queries` (для операцій читання).
- Контролери стали «тонкими»: тепер вони не містять жодної бізнес-логіки (навіть базових перевірок `if/else`), а лише створюють об'єкт Команди або Запиту і передають його у відповідний Handler.

## 2. Переваги CQS: чим допомогло розділення?

- **Single Responsibility Principle (SRP):** Кожен клас тепер має лише одну причину для зміни. Якщо потрібно змінити логіку реєстрації, ми відкриваємо лише `RegisterUserHandler`, не бачачи коду логіну чи оновлення профілю.
- **Незалежна оптимізація потоків:** Ми розділили вимоги до читання і запису. Команди обов'язково проходять через Domain та Repository для захисту інваріантів, тоді як Запити (Queries) можуть звертатися до бази даних (Prisma) напряму, що значно пришвидшує отримання даних.
- **Спрощення тестування:** Тестувати один ізольований Handler набагато простіше, ніж великий Service із десятком залежностей.

## 3. Недоліки CQS: що ускладнилося?

- **Збільшення кількості файлів (Boilerplate):** Тепер для створення однієї сутності потрібно створити два файли (DTO команди та сам Handler) замість одного методу в сервісі.
- **Фрагментація коду:** Щоб зрозуміти всі можливості модуля (наприклад, що взагалі можна зробити з `Task`), тепер потрібно переглянути вміст кількох папок, а не просто відкрити один клас `TaskService`.
- **Складніше налаштування залежностей:** У `Routes` довелося ін'єктувати набагато більше об'єктів (кожен Handler окремо).

## 4. Чим підхід з Command / Query Handler відрізняється від Service?

Service — це універсальний інструмент («швейцарський ніж»), який об'єднує в собі всі операції для певної сутності. Через це він часто стає "God Object" (божественним об'єктом) із роздутим конструктором, куди передається безліч репозиторіїв та інших сервісів.
Handler — це вузькоспеціалізований інструмент («скальпель»). Він інкапсулює логіку виконання лише **однієї** конкретної операції і має залежності лише від тих компонентів, які йому дійсно необхідні для цієї дії.

## 5. Як CQS впливає на розширюваність?

CQS ідеально підтримує **Open-Closed Principle (OCP)**.

- **У підході з Service:** Щоб додати нову фічу (наприклад, «Архівувати задачу»), довелося б змінювати існуючий файл `TaskService`, що несе ризик випадково зламати інші методи.
- **У підході з CQS:** Ми взагалі не чіпаємо існуючий код. Ми просто створюємо новий файл `ArchiveTaskHandler`. Старий перевірений код залишається в повній безпеці. Це робить систему надзвичайно легкою для горизонтального масштабування.

## 6. Чи відрізняється структура даних, яку повертає Query, від доменної моделі?

Так, і це ключова особливість патерну. Запит повертає **Read Model** (через DTO), яка оптимізована під потреби клієнта (фронтенду).

- **Чому це важливо?** Доменна модель (`User` або `Task`) містить бізнес-правила та може мати чутливі дані (наприклад, `passwordHash`). Клієнту на фронтенді ці дані не потрібні. Читаючи дані безпосередньо в оптимізовану Read Model, ми уникаємо "over-fetching" (отримання зайвих даних) і не «світимо» структуру нашої бази даних назовні.
